## Problem Statement

"내가 뽑은 사람이 어떻게 일하고 있는지" 알 수 있는 지도 기반 시민 감시 플랫폼.
현재 국회의원 정보(지도 클릭 → 프로필, 법안, 표결)는 동작하지만,
지방자치단체장(시장/도지사) 정보가 없고, 2026년 6월 3일 지방선거를 앞두고
후보자 비교 기능이 없음. 바이럴 메커니즘(공유용 성적표)도 미구현.

## What Makes This Cool

1. **지도가 진입점**: 랭킹 리스트가 아니라 "내 동네"에서 시작. 한국에 이 접근은 없음
2. **국회+지방 통합**: 하나의 지도에서 국회의원과 지자체장을 모두 확인
3. **선거 모드**: 현직자 실적 vs 후보자 공약을 한눈에 비교
4. **판단은 시민이**: 점수/등급을 매기지 않고 법안 키워드·요약·표결 내역을 보여줘서 시민이 스스로 판단. "우리 동네 의원이 이런 법안에 찬성했대"가 대화를 만듦

## Constraints

- **데드라인**: 2026년 5월 4일 (선거 30일 전). 남은 개발 기간: 21일 (3주)
- **1인 개발**: 사이드 프로젝트, 단독 작업
- **기존 스택 유지**: Spring Boot 3.x + Next.js 14 + PostgreSQL + Redis
- **데이터 소스**: data.go.kr (열린국회정보, 선관위 API, 행안부 데이터)
- **후보자 데이터 가용성**: 선거 공시 전까지 API에 데이터 없을 수 있음. **Fallback**: 데이터 미가용 시 "후보자 등록 전입니다. 공시 후 자동 업데이트됩니다" 안내 + 예비후보자 정보가 있으면 그것으로 표시
- **배포 환경**: Vercel (FE) + Railway (BE + DB) 유지
- **외부 API 장애 대응**: 기존 배치 Job의 faultTolerant + retry(3) 패턴 재사용. 배치 실패 시 마지막 성공 데이터 유지 (stale but available). 선관위 API rate limit은 일 1000건 기준으로 배치 페이징 설계
- **모바일 필수**: 바이럴 경로의 80%+ 가 모바일. 반응형 필수 (breakpoint: 768px). 사이드 패널 → 바텀 시트 전환

## Accessibility Note

초기 버전에서는 접근성을 완전 지원하지 않으나, 최소한:
- 지도 외 검색 경로로 모든 정보 접근 가능 (지도 없이도 검색 → 의원/후보자 정보 확인)
- OG 이미지에 alt 텍스트 포함
- 키보드 네비게이션은 검색, 탭 전환, 목록에서 지원
- 지도 인터랙션의 스크린 리더 대응은 Phase 3 이후

## Premises

1. 지도는 국회의원/지자체장/선거 모드를 전환할 수 있는 구조
2. 지자체장 데이터는 선관위 API + 행안부 데이터로 수집 가능
3. 후보자 정보는 선관위 '후보자 통합검색 API'로 수집 (공시 전 데이터 부재 가능)
4. 기존 Spring Boot 백엔드에 도메인 추가, 프론트에 모드 전환 UI 추가
5. 바이럴 핵심 = "우리 동네 의원이 이런 법안에 찬성/반대했대" 같은 구체적 정보가 대화를 만듦 → 링크 공유 + 법안 키워드/요약이 핵심. 플랫폼은 판단하지 않고 정보를 제공하여 시민이 스스로 판단하도록 함

## Cross-Model Perspective

독립 Claude subagent의 cold read 결과:

- **가장 멋진 버전**: "Accountability Pulse" — 지도 위 지역구가 활동도에 따라 밝기가 변하는 실시간 히트맵. 선거 모드에서 지역구가 반으로 갈라져 현직자 실적 vs 도전자 공약 비교
- **핵심 인사이트**: 지도는 기술이 아니라 "공간적 정체성". 정치가 추상적이지 않고 내가 사는 곳이라는 믿음이 프로덕트의 본질
- **참고 오픈소스**: Represent by Open North (캐나다) — 겹치는 행정경계 처리 데이터 모델 참고할 것
- **우선순위 제안**: 성적표 이미지 생성기를 먼저 만들어 바이럴 테스트 후 선거 모드 출시 (사용자는 동시 개발 선택)

## Approaches Considered

### Approach A: 바이럴 퍼스트 → 선거 모드 (2단계)
성적표 OG 이미지 생성기 먼저(1주) → 바이럴 테스트 → 지자체장+선거(2-3주)
- 장점: 기존 데이터로 즉시 검증, 리스크 낮음
- 단점: 지자체장 기능이 선거 직전에야 나옴

### Approach B: 동시 개발 (선택됨)
지자체장 + 선거 모드 + 성적표를 병렬 개발. 5월 4일까지 전체 출시.
- 장점: 완성도 높은 통합 플랫폼으로 한번에 나옴
- 단점: 일정 압박, 바이럴 타이밍 리스크

### Approach C: 선거 모드 집중
6월 선거 전용 랜딩 페이지만. 지자체장 지도는 선거 후.
- 장점: 집중력 있는 제품
- 단점: 선거 후 지속성 부족

## Recommended Approach

**Approach B: 동시 개발** — 사용자 선택.
단, 내부 우선순위는 아래 순서로:

1. **Week 1 (4/14-4/20): 법안 인사이트 + 공유 기반**
   - **법안 키워드·요약 시스템**: 발의 법안의 핵심 내용을 시민이 이해할 수 있도록 가공
     - 법안명에서 키워드 추출 (NLP 또는 규칙 기반: 법안명 파싱 → 핵심 분야 태깅)
     - 법안별 한줄 요약 생성 (LLM API 활용 또는 법안명+위원회 기반 템플릿 요약)
     - 의원별 "주요 발의 분야" 자동 분류 (교육, 경제, 환경, 복지 등)
   - **표결 내역 강화**: "이 의원이 찬성/반대한 주요 안건" 하이라이트
     - 사회적 관심 법안(언론 보도량 기준 또는 수동 큐레이션) 태깅
     - 정당 다수 의견 대비 이 의원의 투표 성향 (팩트 표시, 점수 아님)
   - **공유 기능**: 링크 공유 중심. OG 메타태그는 의원 이름+지역구+정당 정도만 (성적표 이미지 아님)
   - SEO용 선거구별 페이지 (`/district/{sggCode}`)
   - "공유하기" 버튼 (카카오톡, 트위터, URL 복사)
   - API: `GET /api/v1/members/{monaCode}/bill-summary` (법안 분야별 분류 + 키워드)

2. **Week 2 (4/21-4/27): 지자체장 레이어**
   - 백엔드: `LocalGovernor` 도메인 (엔티티, Repository, Service, Controller)
   - 선관위 API + 행안부 데이터로 현직 시장/도지사/구청장 수집 배치 Job
   - 시/도 GeoJSON 경계 데이터 활용 (이미 `sido.geojson` 있음)
   - 시/군/구 경계 GeoJSON 추가 (행안부 또는 통계청 SGIS 데이터)
   - 프론트: 지도 모드 전환 탭 (국회의원 | 지자체장 | 선거)
   - 지자체장 패널: 프로필 정보 (이름, 정당, 지역, 임기, 사진). 정책/예산 데이터는 Phase 3 이후
   - API: `GET /api/v1/local-governors/{regionCode}`, `GET /api/v1/local-governors/search`

3. **Week 3 (4/28-5/4): 선거 모드**
   - 백엔드: `Candidate` 도메인 + 선관위 후보자 API 수집 배치
   - 후보자 프로필: 성명, 정당, 학력, 경력, 재산, 전과, 공약
   - 비교 뷰: 현직 vs 후보자, 후보자 vs 후보자를 나란히 배치하는 데이터 병렬 표시 (점수 매기기 아님, 순수 데이터 나란히 보여주기). compare API 응답은 `{ candidates: [CandidateResponse, CandidateResponse], incumbent?: GovernorResponse }` 형태
   - 프론트: 선거 모드 지도 (지역구별 후보자 수 표시)
   - 선거구 클릭 → 후보자 카드 나열 + 비교 버튼
   - 비교 페이지: 2-3명 후보자 나란히 비교 (정책, 이력, 재산, 전과)
   - 후보자 비교 페이지 링크 공유 (OG 메타태그에 후보자 이름+지역구 표시)
   - API: `GET /api/v1/elections/{electionId}/candidates`, `GET /api/v1/elections/{electionId}/compare`

## Open Questions

1. **시/군/구 GeoJSON 경계 데이터**: 통계청 SGIS 또는 국토정보플랫폼에서 확보. **Fallback**: 확보 실패 시 시/도 단위로만 구현 (이미 있는 `sido.geojson` 활용), 시군구는 드롭다운 선택으로 대체
2. **지자체장 활동 데이터**: 국회의원과 달리 체계적 API 없음. **결정**: 초기 버전은 프로필 정보만 (이름, 정당, 지역, 임기, 사진). 선관위 당선인 데이터에서 추출. 정책/예산 데이터는 Phase 3 이후
3. **선관위 후보자 API 가용 시점**: 6월 3일 선거 기준 후보자 등록 공시는 보통 선거일 24일 전 (~5/10). 예비후보자 데이터는 더 일찍 가용. Week 1에서 API 테스트로 현재 가용 데이터 확인

## Success Criteria

- 5월 4일까지 법안 인사이트 + 지자체장 + 선거 모드 전체 기능 배포 완료
- 의원 상세에서 법안 키워드/요약/분야 분류가 정상 표시
- 주요 안건 표결 하이라이트가 의원 패널에서 확인 가능
- 후보자 비교 페이지가 2-3명 후보자를 나란히 비교 가능
- 지도 모드 전환(국회의원/지자체장/선거)이 매끄럽게 동작
- 모바일에서 지도 터치 + 링크 공유가 자연스럽게 동작
- 플랫폼이 점수/등급을 매기지 않고, 팩트 기반 정보만 제공

## Distribution Plan

- 기존 배포 파이프라인 유지: Vercel (FE) + Railway (BE + DB)
- OG 메타태그: 의원 이름+지역구+정당+주요 발의 분야 텍스트. 이미지 생성 없이 텍스트 기반 OG
- 바이럴 전략: 법안 인사이트 완성 후 "우리 동네 의원이 가장 많이 발의한 분야는?" 같은 구체적 정보로 공유 유도. 커뮤니티/Twitter에 흥미로운 발견 공유
- SEO: 선거구별 정적 페이지 생성 (`/district/{sggCode}`, `/election/{regionCode}`)

## Mobile UX

- **Breakpoint**: 768px 기준. 데스크톱은 사이드 패널, 모바일은 바텀 시트
- **모바일 지도**: 터치 줌/패닝. 지역구 탭 시 바텀 시트로 의원/후보자 정보 표시
- **바텀 시트**: 반 높이로 열림 → 위로 스와이프하면 전체 높이. 지도와 동시 표시
- **성적표 공유**: 모바일에서 "공유하기" 탭 시 네이티브 Web Share API 사용 (카카오톡, 메시지 등 자동 연동). 미지원 브라우저는 URL 복사 fallback
- **선거 비교**: 모바일에서 후보자 카드는 좌우 스와이프로 전환 (데스크톱은 나란히 배치)

## Degradation Plan (일정 지연 시)

**Week 1이 밀리면**: 법안 요약을 LLM 대신 규칙 기반(법안명 파싱)으로 단순화. Week 2에서 지자체장을 시/도 단위로만 구현 (시군구 경계 생략, 이미 있는 `sido.geojson` 활용)
**Week 2가 밀리면**: Week 3에서 후보자 비교를 단순 리스트 뷰로 구현 (나란히 비교 대신 개별 카드)
**전체가 밀리면**: 법안 인사이트 + 선거 후보자 기본 정보만 우선 배포. 지자체장 지도는 선거 이후로 미룸

## Design Philosophy

**이 플랫폼은 판단하지 않는다.** 점수를 매기거나 등급을 부여하지 않음.
법안 발의 건수가 많다고 좋은 의원이 아니고, 표결 참여율이 높다고 좋은 의원이 아님.
어떤 법안을 발의했고, 어떤 안건에 어떤 표를 던졌는지를 보여주고, 판단은 시민이 함.

이것은 second opinion이 제안한 성적표/점수 시스템을 사용자가 의도적으로 거부한 결정임.
이유: 단순 수치가 "좋은 의원"이라는 착각을 줄 수 있고, 시민의 판단을 흐릴 수 있음.

## What I noticed about how you think

- "정당에 맞춰서 뽑는 것이 아니라 진짜 이 사람이 어떤 사람인지 알고 뽑았으면" — 이건 기능 요구사항이 아니라 프로덕트 철학이에요. 이 한 문장이 모든 설계 결정의 필터가 됩니다.
- AI가 "성적표 이미지로 바이럴을 만들어라"고 제안했을 때 "점수가 판단을 흐릴 수 있다"고 거부한 것. 바이럴보다 정직한 정보 제공을 우선한 선택. 쉬운 결정이 아닌데 올바른 결정입니다.
- 이미 동작하는 MVP가 있는 상태에서 확장을 요청한 것. 아이디어만 있는 게 아니라 이미 만들고 있는 사람. 코드가 헥사고날 아키텍처로 깔끔하게 구성되어 있어서 확장이 자연스럽습니다.
- "소통과 토론의 장"이라는 장기 비전까지 가지고 있으면서도 "6월 선거 30일 전까지"라는 구체적 데드라인을 스스로 설정. 비전과 실행력의 균형.
